The problem

AI classification is useful — and probabilistic. The model can return a category that doesn't exist, a confidence score outside 0–1, an empty required field, or nothing at all. Downstream operational systems are not forgiving: a wrong routing decision reaches a sales team, a CRM, or a support queue before anyone notices.

Most AI workflow demos stop at classification. This system is built for the other case — what happens when the model is wrong.

What was built

An 8-stage pipeline that sits between an AI classifier and operational systems: input sanitisation → AI call → strict schema validation → retry with a strict prompt → safe-default fallback → threshold routing → alerting → SQLite audit trail. No field from the AI response is trusted until it passes validation, and every decision — including every failure — is persisted under a traceable run ID.

The failures, on display

Three records in the test set are deliberately poisoned — each seeds a real, observed LLM failure mode. All three were caught by the validator, failed their strict-prompt retry, received the safe default, and routed to human review. None touched a downstream system.

Four more records with no usable AI response — empty, whitespace-only, too-short, or unseeded input — also flowed through fallback to human review: seven failures handled in total. The sanitiser separately stripped injected script tags and their contents from two records before the AI was ever called.

The correction the repo doesn't hide

DOCUMENTATION vs REALITY
Earlier revisions of the README claimed "fallback triggered on 2 records," later "3" — both derived from reading the code. The live run observed 7. The README now states 7, and the history of being wrong is kept in EVAL_RESULTS.md.

The rule this encodes: documentation is corrected by runs — never the other way around.

What this eval does not cover

A credible eval states its own boundaries. From the committed results file, unedited in substance:

KNOWN LIMITATIONS — SELF-DECLARED

  • All results are from simulation mode with pre-seeded responses; live model output is non-deterministic and will differ.
  • MAX_RETRIES = 1 — a single retry; production would need exponential backoff.
  • No load or concurrency testing; SQLite is single-writer.
  • The 2,000-character truncation path exists but is not exercised by this test set.

The decision on record

ADR-001 — VALIDATE AI OUTPUT BEFORE ROUTING · ACCEPTED 2026-06-19

Downstream systems don't validate what they receive; if the router silently mapped an unknown value to an arbitrary decision, the failure would be invisible and untraceable. So the architecture commits to one rule: no field from the AI response is trusted until it passes validation. The full decision record — context, alternatives, consequences — is in the repo.
