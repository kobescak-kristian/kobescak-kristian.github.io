The run that failed its own gate.

keyed eval run · 2026-07-07 · 75 records gate locked 2026-07-06 — before the run repo verified @ 8ad6e07 RESULT: FAIL — PUBLISHED AS-IS

The problem

The AI Context Engine is the explanation layer of the pipeline: it retrieves past cases and rules, grounds each decision in that precedent, and cites its sources. It has two decision paths — a deterministic four-rule fallback that needs no API key, and an LLM layer that reads the retrieved context.

The only reason to route a decision through an LLM is to do better than the rule-based default. That is a measurable claim. And a measurement only means something if the bar is fixed before the result is known — and the result is recorded whichever way it lands.

The gate, committed first

The thresholds live in eval_config.py, locked 2026-07-06 — before the first official eval run. From the committed file: "the numbers are fixed here first, then run_eval.py is pointed at them, so no threshold is chosen after seeing a result."

GateThresholdRationale (from the committed config)
FALLBACK_GATE ≥ 68 / 75 Audit-measured baseline for the deterministic path — a regression floor, not a target.
KEYED_GATE ≥ 69 / 75 Must strictly beat the fallback — the only reason to route through an LLM is to do better than the rule-based default.
KEYED_TARGET 71 / 75 Reported, not gated — no evidence basis existed for treating it as pass/fail.

The ordering is public: in the repo history, the commit that adds these thresholds precedes both result commits. Anyone can check it with git log.

The two runs

Both runs executed on 2026-07-07 against the same committed 75-record eval set. The fallback run passed its gate. The keyed run — the first successful one; the originally hardcoded model had been retired and returned 404 on every request, so the model was swapped before a keyed run could complete — did not pass its gate. It scored below the deterministic baseline it was required to strictly beat.

The miss pattern

Seventeen misses is not the interesting part. The interesting part is that they cluster — two failure modes, not random noise. This analysis is committed in the repo alongside the FAIL.

ClusterMissesWhat the model did
Under-committing on clear-cut cases 8 6× qualify → manual_review, 2× qualify → escalate — high-confidence, clearly-qualifying leads routed to a review queue instead of qualified outright. More conservative than the ground truth.
Avoiding hold 4 4× hold → manual_review — the same category the deterministic fallback structurally cannot produce (none of its four rules emits hold). The LLM has hold in its schema and system prompt, and converges on manual_review anyway. Grounding in retrieved context did not close this gap.
Scattered 5 2× manual_review → escalate, 1× each manual_review → hold / manual_review → disqualify / disqualify → manual_review.

What held: grounding, proven separately

The run failed on accuracy against generated labels. It did not fail on grounding: every keyed row cites the specific retrieved document IDs it reasoned from. A committed side-by-side run of the same lead with and without retrieved context shows the with-context leg citing case_009, rule_006, and case_005 by ID at confidence_adjusted 0.88 versus 0.75 without — the comparison is Example 8 in the repo's EXAMPLE_OUTPUTS.md.

GROUNDING ≠ ACCURACY
Grounding and accuracy are different claims; this repo can now support the first with evidence and cannot support the second.

→ EVAL_RESULTS.md, interpretation section — committed together with the FAIL, not after it was fixed.

What did not happen

The score created an obvious temptation: tune the prompt, re-run, publish the better number. The results file rules it out by name — "doing so after seeing this result would be gate-shopping."

So: single run, no retries, no prompt iteration. The rule is recorded with the result: if the system prompt or model is revised later, that is a new run with its own entry — not a silent re-run. The gate thresholds were never touched after the result; the one post-run edit to the config file changed a line of documentation wording, not a number, which anyone can confirm with git log -p eval_config.py. And the README was reconciled to report the failed run plainly instead of softening it.

What this eval does not claim

A credible eval states its own boundaries. From the committed results file, unedited in substance:

KNOWN LIMITATIONS — SELF-DECLARED

  • The eval labels are synthetic — generated, not real client data. 58/75 and 68/75 describe agreement with a generated label set, not real-world decision accuracy.
  • 75 records is a point estimate, not a statistically powered result.
  • One keyed run, one model, no retries — no comparison against other models was attempted.

The decision on record

ADR-0001 — SEMANTIC RETRIEVAL WITH A TF-IDF FALLBACK · ACCEPTED 2026-07-04

Retrieval attempts a semantic embedding model first; if the model isn't available, it falls back to a self-contained TF-IDF embedder built from the same corpus. Both paths produce same-shaped vectors into the same index, and every result records which mode built it — so the eval runs anywhere, with or without the model download.
