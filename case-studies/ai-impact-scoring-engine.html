CASE STUDY · RECONCILIATION · AI IMPACT SCORING ENGINE

The financial engine that didn’t reconcile with its own ledger.

independent audit · 2026-07-05 · 75 synthetic leads fixes verified public 2026-07-06 audited @ 98f9df6 · current @ b127fc4 RESULT: €3,130 DISCREPANCY — FOUND, FIXED, RECONCILED

The problem

The AI Impact Scoring Engine prices AI routing decisions: revenue won, cost of wasted sales effort, revenue lost to missed leads. Its entire pitch is deterministic, auditable financial numbers — the scoring is rule-based and runs before any AI call, so the same input always produces the same figure, and the LLM can only narrate numbers it is handed, never invent them (ADR-0001, committed in the repo).

A system that sells auditability has to survive being audited. So it was — from its public HEAD, by a separate model session instructed to be hostile: fresh clone, quickstart on a default Windows console, all five API endpoints exercised live, aggregate math reconciled against per-lead outputs.

What the audit could not break

Every numeric claim in the README reproduced exactly on a live run — 75 leads, all revenue figures, all three rates, both worked examples verbatim (audit claim-verification table: 25 claims, the README Outcome section verified line by line; full report is an internal record). The quickstart ran exit-0 with no encoding workarounds. The git history matched the version log. The layer separation held: the aggregation module imports no LLM client, and the AI layer’s function signature accepts final metrics only — it cannot see raw lead data.

What it found instead

One level deeper: the flagship aggregate did not reconcile with the system’s own per-lead ledger.

Pre-fix figures from the 2026-07-05 audit run (internal record); the pre-fix README net of €116,110 is in the repo’s public git history.

Summing GET /impact/{lead_id} over all 75 leads gave €119,240. GET /impact reported net impact €116,110. The €3,130 difference was the delayed-conversion penalty, counted twice — once inside per-lead revenue (entered net of penalty) and again as an itemized loss. Each delayed lead effectively carried a 20% penalty while the config and the README both said 10%. A reviewer who cross-sums the per-lead endpoint catches it in minutes; the AI layer narrated the inconsistent numbers with confidence.

Two more findings compounded it for the same skeptical reader: a valid input combination the README explicitly permits — archived + converted_delayed — fell out of the routing table and silently returned unknown with €0 impact, while the ADR asserted every valid pair was mapped (14 of 15 were); and the aggregate layer hardcoded penalty constants instead of reading the config multipliers the documentation pointed at, so tuning the config moved per-lead figures but not the headline.

The fixes, in public history

Two remediation commits, both in the open history and dated in the README version log. The accounting decision (gross-revenue entry, penalty counted once) was made by the owner; the fixes were implemented in-session and re-verified from a fresh public clone on 2026-07-06 before the record closed (internal record; the fix trail itself is public).

After the fix, net impact reconciles exactly with the sum of per-lead financial_impact: both €119,240 on the demo dataset (README version log, verifiable by re-running seed_and_run.py and summing the per-lead endpoint). Revenue generated changed to €168,300; revenue lost (€49,060) and every rate (41.3% / 45.2% / 44.4%) unchanged — the fix moved only what the accounting error had distorted.

The before/after is reconstructable by anyone: the pre-fix README in git history says €116,110; the current README says €119,240; the delta is the double-counted penalty.

What this does not claim

All figures come from a 75-record synthetic dataset intentionally constructed to show both strong and weak decision performance — no production deployment is claimed. The README’s own Known Limitations carry the rest:

KNOWN LIMITATIONS — SELF-DECLARED

  • Financial multipliers (15% / 10% / 100%) are assumed constants, not calibrated from real sales-cycle costs.
  • The missed-opportunity figure — the largest loss number in the demo — depends on outcome feedback that real systems rarely have complete, so it is the least reliable metric.
  • Recommendations are not anchored to actual upstream thresholds.
  • There is no coverage gate, so low-outcome-coverage runs look as confident as full ones.

Of seven minor audit findings, one was fixed in remediation; six were carried forward, documented rather than silently fixed (internal record).

The decision on record

ADR-0001 — DETERMINISTIC SCORING BEFORE THE AI LAYER · ACCEPTED 2026-07-04

Financial numbers are computed by static lookup and fixed multipliers before any model runs; the only LLM module receives the finished metrics object and nothing else. Consequence stated in the ADR itself: hallucination risk is bounded to prose — the model cannot fabricate or alter a euro figure. The audit tested that boundary and it held; what broke was arithmetic in the deterministic layer, which is exactly the layer a reviewer can catch — and did.
